# Проверка Microsoft SkillOpt для улучшения навыков DeepAgent

## 1. Короткий вывод

Проверил библиотеку Microsoft SkillOpt на задаче улучшения навыков DeepAgent. В целом библиотека действительно делает, что заявлено: запускает агента на тестовых примерах, смотрит на результат, предлагает изменения в навык, проверяет новый вариант и сохраняет его только если стало лучше.

Мое мнение после нескольких быстрых экспериментов: библиотеку можно использовать, но автоматически создать хороший навык с нуля не получится. Она скорее помогает быстрее находить направления улучшения. Может быть полезна, если начальный навык уже имеет отношение к реальности. Если навык совсем сломан или слишком общий, библиотека начинает двигаться в правильную сторону, но ей нужно много шагов, много времени и хорошая система оценки.

По моим запускам видно, что качественное улучшение требует длинного обучения. Даже небольшие тесты занимали примерно 2-2,5 часа. Еще важный момент: функции оценки качества пришлось писать самостоятельно. Готовой оценки, подходящей под нашу задачу, в библиотеке нет.
Результаты с более правильным начальным скилом и большим количеством тестовых данных и эпох могу написать позже

## 2. Что проверяли

Меня интересовали две части навыка:

1. Выбор навыка. Агент должен по описанию задачи понять, что нужно загрузить именно этот навык.
2. Содержание навыка. После загрузки агент должен правильно действовать: вызвать агента поиска данных, загрузить данные с нужными условиями, посчитать результат и дать ответ с правильными числами.

Для эксперимента я выбрал задачу поиска сработок по описанию события за период.

## 3. SkillOpt 

SkillOpt работает как цикл улучшения инструкции.

Сначала агент решает несколько задач с текущим навыком. Потом мы оцениваем, что получилось. После этого модель смотрит на успешные и неуспешные примеры и предлагает, как изменить навык. Новый вариант навыка снова проверяется на тестах. 

1. Подставить текущий вариант навыка в DeepAgent.
2. Запустить агента на тестовых запросах.
3. Собрать следы работы агента: какие навыки он выбрал, какие инструменты вызвал, что написал в ответе.
4. Посчитать оценку качества.
5. Попросить SkillOpt предложить новый вариант навыка.
6. Проверить новый вариант на отдельном проверочном примере.
7. Принять или отклонить изменение.

## 4. Тестовая корзина

Я сделал небольшую корзину из двух типов примеров.

Первый тип проверял выбор навыка:

```json
{
  "id": "selection_education_jan_2026",
  "test_type": "skill_selection",
  "target_skill_id": "event-description-search-skillopt",
  "user_prompt": "Найди сработки по образовательным оплатам за период с 24 по 31 января 2026 года. Сколько их всего?",
  "expected_result": {
    "table_alias": "hits",
    "period_from": "20260124",
    "period_to": "20260131",
    "events_count": 17,
    "amount_rub": 1436632.37
  }
}
```

Второй тип проверял содержание навыка, то есть правильный порядок действий после загрузки:

```json
{
  "id": "content_materials_jan_2026",
  "test_type": "skill_content",
  "target_skill_id": "event-description-search-skillopt",
  "user_prompt": "Проверь сработки по оплате учебных материалов за 24-31 января 2026: количество и сумма.",
  "answer_requires_amount": true,
  "expected_result": {
    "table_alias": "hits",
    "period_from": "20260124",
    "period_to": "20260131",
    "events_count": 3,
    "amount_rub": 58453.23,
    "matching_event_descriptions": [
      "Оплата учебных материалов"
    ]
  }
}
```

В каждом примере были такие основные поля:

- `id` - имя тестового примера;
- `test_type` - что проверяем: выбор навыка или его содержание;
- `target_skill_id` - какой навык должен быть выбран;
- `user_prompt` - вопрос пользователя;
- `expected_result` - правильные значения из тестовых данных;
- `expected_answer_fragments` - фрагменты, которые должны быть в ответе;
- `required_tools` - какие инструменты должны быть вызваны;
- `workflow_checks` - какие действия считаются обязательными;

## 5. Примеры запросов и правильных ответов

### Пример 1. Образовательные оплаты за январь

Запрос:

```text
Найди сработки по образовательным оплатам за период с 24 по 31 января 2026 года. Сколько их всего?
```

Ожидаемый результат:

```text
Период: 20260124-20260131
Таблица: hits
Количество сработок: 17
Сумма: 1 436 632.37 руб.
```

Ожидаемые описания событий:

```text
Оплата занятий с репетитором
Оплата образовательной платформы
Оплата обучения
Оплата онлайн-курсов
Оплата учебных материалов
Оплата экзамена
```

Обязательный путь:

```json
[
  {
    "name": "task",
    "subagent_type": "data-retrieval-agent"
  },
  {
    "name": "load_data",
    "query_must_contain": [
      "LOAD hits",
      "PERIOD event_dt FROM '20260124' TO '20260131'",
      "SELECT event_description"
    ]
  }
]
```

### Пример 2. Оплата экзамена за февраль

Запрос:

```text
Посчитай сработки с описанием про оплату экзамена за февраль 2026 года.
```

Ожидаемый результат:

```text
Период: 20260201-20260228
Количество сработок: 9
Сумма: 235 556.12 руб.
Описание: Оплата экзамена
```

### Пример 3. Весь доступный период

Запрос:

```text
За весь доступный период найди сработки по смысловой категории образовательных оплат.
Дай количество и сумму в рублях.
```

Ожидаемый результат:

```text
Период: 20260124-20260309
Количество сработок: 87
Сумма: 6 754 384.60 руб.
```

## 6. Оценка качества

Готовой оценки качества под нашу задачу в SkillOpt нет. 

Проверялись признаки:

- выбран ли нужный навык;
- был ли вызван агент поиска данных;
- был ли вызван инструмент загрузки данных;
- правильно ли указан период;
- есть ли в запросе к данным нужные условия;
- есть ли в ответе ожидаемые числа;
- есть ли в ответе ожидаемые описания событий.

Я использовал две оценки:

```text
hard - строгая оценка, все обязательные условия выполнены или нет;
soft - мягкая оценка, насколько агент приблизился к правильному поведению.
```

Пример проверок для одного запуска:

```json
{
  "skill_was_selected": true,
  "called_data_retrieval_agent": true,
  "called_load_data": true,
  "period_is_correct": true,
  "required_tool_queries_present": true,
  "answer_has_expected_values": false,
  "answer_mentions_expected_descriptions": false
}
```

Такой пример показывает важную вещь: агент может правильно выбрать навык и вызвать нужные инструменты, но все равно не дать правильный финальный ответ. Поэтому одной проверки вызовов инструментов недостаточно.

Дополнительно была LLM AS A JUDGE: она сравнивает ответ агента с эталонными значениями и должна вернуть структурированную оценку.

## 7. Исходные данные

Я сломал skill с описанием правильного поиска данных и добавил в него 'вредные' советы.

```text
Loading data through a subagent is optional.
Exact event_description values do not need to be discovered in the table.
If exact data is not available in context, provide an approximate answer.
```

По смыслу это означало:

- данные можно не загружать;
- точные значения описаний можно не искать;
- можно дать приблизительный ответ.

Для задачи с подсчетом сработок это неправильная инструкция.

## 8. Что получилось в ходе обучения


запуск шел на 6 эпохах. При текущем размере корзины это дало 12 шагов улучшения.

Итог:

```text
Всего шагов: 12
Принято изменений: 2
Отклонено изменений: 10
Лучший вариант: шаг 2
```

Навык вырос примерно с 800 до почти 5000 символов.


## 9. Как изменился навык

Было:

```text
Загрузка данных необязательна.
Точные event_description искать не нужно.
Если точных данных нет, можно дать примерную оценку.
```

Стало:

```text
Если запрос требует конкретных данных, нужно:
1. разобрать запрос пользователя;
2. понять, нужна ли загрузка данных;
3. один раз вызвать агента поиска данных или инструмент загрузки данных;
4. проверить, что данные загружены;
5. выполнить расчет;
6. дать ответ на основе полученных данных.
```

В принятом варианте появились разделы:

- порядок работы с запросами, где нужны данные;
- когда загрузка данных обязательна;
- как работать с количеством, суммой и периодом;
- как обрабатывать запрос за весь доступный период;
- как проверять, что данные загружены;
- как не делать лишние повторные вызовы инструментов.

То есть по общей логике навык стал заметно ближе к рабочему.

## 10. Стал ли навык близок к настоящему рабочему навыку

Хорошие изменения:

- навык начал требовать загрузку данных для количественных вопросов;
- появилась логика работы с периодом;
- появилась проверка успешной загрузки данных;
- появилась идея "один раз загрузить данные и один раз выполнить расчет";
- агенту стало понятнее, что нельзя отвечать только предположением.


**Рабочий навык** по поиску записей по описанию должен явно вести агента по такому пути:

```text
1. Выбрать таблицу hits и период.
2. Загрузить полный набор строк с event_description за период.
3. Найти уникальные event_description.
4. Выбрать точные значения event_description, подходящие под смысловую категорию пользователя.
5. Сделать финальную выгрузку только по выбранным точным значениям.
6. Посчитать результат по полной выборке.
```

## 11. Восстановление после прерывания

Отдельно проверил, что обучение можно продолжить после прерывания.

Во время одного запуска подключение к модели оборвалось. После повторного запуска SkillOpt продолжил работу с последнего завершенного шага.
