ПЛАН РАБОТ: ПЕРВЫЕ ТРИ ЭТАПА РАЗВИТИЯ DEEPAGENT

Область плана
=============

Этот документ описывает первые три этапа:

1. Надёжное выполнение.
2. Наблюдаемость и трассировка.
3. Eval-корзина.

Память, learned skills, Better Harness, SkillOpt и self-learning loop начинаются
только после получения стабильного и измеримого baseline.


Архитектурные ограничения
=========================

1. Агент остаётся построенным на публичных механизмах DeepAgents:
   create_deep_agent, HarnessProfile, middleware, subagents, skills, backend
   и checkpointer.

2. Собственный LangGraph-граф не создаётся.

3. LLM принимает смысловые решения:
   - выбирает релевантные skills;
   - определяет необходимость делегирования;
   - строит план;
   - выбирает инструменты;
   - интерпретирует результаты.

4. Кастомный код отвечает только за:
   - сборку harness;
   - передачу контекста;
   - безопасность;
   - валидацию формата;
   - предотвращение технических циклов;
   - хранение артефактов;
   - трассировку и измерение.

5. Не устанавливаются жёсткие ограничения на число skills, файлов или memory
   records, которые может выбрать LLM. Модель должна загрузить достаточный объём
   контекста для задачи.

6. Progressive disclosure задаётся инструкциями и структурой файлов, а не
   скрытым top-k или обрезанием списка выбранных skills.

7. При ошибке skill routing запрещён fallback на загрузку всех skills.
   Ошибка должна быть видимой и обрабатываться как ошибка выбора контекста.

8. Любая неочевидная автоматическая операция должна отражаться в trace.



ЭТАП 0. ПОДГОТОВКА BASELINE
============================

Цель
----

Зафиксировать текущее поведение до изменения harness, чтобы отличать улучшения
от регрессий.

Задачи
------

[BASE-01] Выбрать 10-15 репрезентативных сценариев:
- простой вопрос без данных;
- одна выборка из таблицы;
- semantic search по event_description;
- обработка большого pickle;
- вычисление через execute_python_code;
- отсутствие подходящего skill;
- ошибка load_data;
- пустой результат;
- неоднозначный период;
- повторный вызов инструмента.

[BASE-02] Для каждого сценария сохранить:
- user request;
- ожидаемые источники;
- ожидаемые обязательные действия;
- запрещённые действия;
- ожидаемый результат;
- допустимые ограничения ответа.

[BASE-03] Зафиксировать исходные метрики:
- успешность;
- число model calls;
- число tool calls;
- повторные вызовы;
- максимальный размер контекста;
- длительность;
- выбранные skills;
- наличие неподтверждённых фактов.

[BASE-04] Сохранить версию конфигурации, prompts и списка tools, с которой
снят baseline.

Результаты этапа
----------------

- eval fixtures для baseline;
- baseline report;
- перечень известных failure modes;
- исходные количественные метрики.

Критерий завершения
-------------------

Каждый последующий вариант harness можно сравнить с исходной версией на одном
и том же наборе сценариев.


ЭТАП 1. НАДЁЖНОЕ ВЫПОЛНЕНИЕ
============================

Цель
----

Уменьшить ошибки слабой модели за счёт ясного контекста, небольшого набора
релевантных инструментов, явных контрактов и устранения технических циклов.
Не переносить предметные решения из LLM в скрытый Python-код.


1.2. Состав инструментов
------------------------

[REL-06] Отключить general-purpose subagent.

[REL-07] Сохранить data-retrieval-agent как специализированного subagent.

[REL-08] Supervisor должен получать минимальный универсальный набор:
- task;
- execute_python_code;
- load_skills;
- write_todos

[REL-09] Data-retrieval-agent должен получать:
- load_data;
- execute_python_code;
- необходимые DeepAgents filesystem tools для skills и artifacts.

[REL-10] Устранить дубли:
- glob и glob_search;
- grep и grep_search;
- другие инструменты с одинаковой функцией.

[REL-11] Не скрывать инструменты на основании неявной классификации бизнес-задачи.
Статическое исключение допустимо только для заведомо ненужных или дублирующих
инструментов.

[REL-12] Оставить execute_python_code доступным для:
- обработки artifacts;
- вычислений;
- получения текущего времени через разрешённый Python runtime;
- просмотра разрешённого runtime-состояния;
- создания итоговых файлов.


1.3. Skill routing
------------------

[REL-13] Сохранить обязательный LLM-выбор релевантных skills.

[REL-14] Результат selector должен разрешать выбор от нуля до любого числа
skills. В schema и prompt не должно быть max_skills или top_k.

[REL-15] Удалить текущий fallback:

except Exception:
    return all_paths

[REL-16] При ошибке structured output выполнить не более одного корректирующего
LLM-вызова:
- передать исходный index;
- объяснить ошибку формата;
- потребовать использовать только существующие пути.

[REL-17] Если повторный выбор неуспешен:
- вернуть пустой выбор;
- записать status=selection_failed;
- передать supervisor краткое описание ошибки;
- не загружать все skills автоматически.

[REL-18] Если LLM осознанно выбрала ноль skills, различать:
- selection_status=success;
- selected_paths=[];
- selection_reason.

[REL-19] Валидировать только технический контракт:
- путь существует в index;
- нет выдуманных путей;
- нет дублей.

Валидатор не должен решать, какой skill семантически правильный.

[REL-20] Не передавать полный Skills Index в system prompt на каждом model call.
После LLM-routing передавать выбранные skills. Дозагрузка выполняется через
load_skills по запросу модели.

[REL-21] Рассмотреть изменение load_skills так, чтобы он мог принимать
естественно-языковую потребность и запускать тот же LLM selector. При этом
сохранить возможность явно передать известные имена skills.


1.4. Prompt и контекст
----------------------

[REL-22] Устранить дублирование между:
- DeepAgents base prompt;
- SYSTEM_PROMPT;
- tool description overrides;
- AGENTS.md;
- SKILL.md.

[REL-23] Оставить в supervisor prompt только:
- роль;
- границы ответственности;
- порядок источников истины;
- правила делегирования;
- правила работы с evidence;
- требования к финальному ответу.

[REL-24] Перенести domain workflows только в skills.

[REL-25] Сократить descriptions tools:
- назначение;
- вход;
- выход;
- когда использовать;
- одна-две критические ошибки.

[REL-26] Удалить длинные демонстрационные примеры из постоянно передаваемого
контекста. Примеры должны находиться в skill references и загружаться при
необходимости.

[REL-27] max_chars_per_skill оставить конфигурируемым. Не использовать его как
скрытый лимит числа skills. Если файл превышает технический предел контекста,
возвращать явную информацию о truncation.

1.5. Tool loop и retries
------------------------

[REL-29] Изменить ToolLoopGuard:
- учитывать имя tool;
- учитывать нормализованные аргументы;
- различать точный повтор и исправление.

[REL-30] Блокировать технический цикл одинаковых вызовов, но не запрещать
модели загрузить дополнительные skills или файлы, когда появились новые
основания.

[REL-31] Для ошибки tool возвращать:
- error category;
- failing input;
- expected/observed;
- retryable;
- краткую correction hint.

[REL-32] Retry выполнять только при изменении аргументов или при явно transient
ошибке. Одинаковый вызов не должен автоматически повторяться.

[REL-33] Пустой результат load_data не считать технической ошибкой. Он должен
быть передан модели вместе с использованными source, period и filters.


1.6. Контракт subagent
----------------------

[REL-34] Supervisor передаёт data-retrieval-agent:
- business objective;
- known inputs;
- выбранные skills;
- period;
- ожидаемый формат отчёта;
- stopping condition.

[REL-35] Supervisor не передаёт придуманные SQL, WHERE, таблицы или фильтры.

[REL-36] Subagent возвращает компактный report:
- result;
- sources;
- filters;
- tool calls;
- artifacts;
- limitations;
- missing data.

[REL-37] Supervisor сохраняет финальный synthesis.


Критерий завершения этапа 1
---------------------------

- general-purpose отключён;
- нет fallback на все skills;
- LLM может выбрать произвольное число skills;
- число дублирующихся tools равно нулю;
- технические циклы одинаковых tool calls блокируются;
- исправленные вызовы разрешены;
- execute_python_code сохранён;
- основные baseline-сценарии не дают неподтверждённых ответов;
- все решения selector и профиля видны в trace.


ЭТАП 2. НАБЛЮДАЕМОСТЬ И ТРАССИРОВКА
====================================

Цель
----

Сделать каждый запуск объяснимым и измеримым без чтения полного raw prompt или
огромного текстового лога.


2.1. Формат событий
-------------------

[OBS-01] Создать типизированную схему TraceEvent.

Обязательные поля:
- event_id;
- parent_event_id;
- run_id;
- thread_id;
- timestamp;
- event_type;
- agent_name;
- model_name;
- status;
- duration_ms;
- payload;

[OBS-02] Ввести события:
- run_started;
- harness_profile_selected;
- model_call_started;
- model_call_completed;
- context_selection_started;
- context_selection_completed;
- skill_loaded;
- tool_call_started;
- tool_call_completed;
- subagent_started;
- subagent_completed;
- artifact_created;
- context_compacted;
- run_completed;
- run_failed.

[OBS-03] Не записывать скрытые рассуждения модели. Сохранять только observable
messages, tool calls, выбранный контекст и результаты.


2.2. Структура run
------------------

[OBS-04] Перейти к структуре:

runs/<run_id>/
    trace.jsonl
    summary.json
    artifacts/

[OBS-05] trace.jsonl является append-only журналом.

[OBS-06] summary.json содержит:
- user request hash или текст согласно настройкам privacy;
- final status;
- profile;
- selected skills;
- subagents;
- model/tool call counts;
- token usage;
- latency;
- artifacts;
- errors;
- final answer summary.

[OBS-07] Связать tool output artifact с конкретным tool event.


2.3. Инструментирование
-----------------------

[OBS-08] Обернуть model calls middleware, фиксирующим:
- количество входных сообщений;
- приблизительный или фактический token count;
- список доступных tools;
- размер system prompt;
- finish reason;
- usage metadata.

[OBS-09] Обернуть tool calls:
- имя;
- args после redaction;
- compact result;
- success/error;
- artifact paths;
- duration.

[OBS-10] Отдельно фиксировать skill routing:
- переданный index;
- выбранные paths;
- selection reason;
- validation errors;
- retry;
- итоговый status.
