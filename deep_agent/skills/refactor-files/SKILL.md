# Рефакторинг файлов

Используй этот skill, когда пользователь просит отрефакторить существующий код или другие исходные файлы проекта.

## Рабочий цикл

1. Проанализируй текущий код до правок:
    - какие данные, функции, классы, типы и публичные контракты уже есть;
    - как код вызывается и в какой последовательности выполняется;
    - какие inputs, outputs, exceptions и side effects нужно сохранить;
    - какие есть текущие комментарии и пометки автора;

2. Вноси изменения в порядке использования кода:
    - сначала imports, constants и настройки;
    - затем схемы данных и типы;
    - затем helper-функции;
    - затем основную бизнес-логику;

3. Сохраняй поведение пользователя:
    - не меняй публичные имена, названия переменных, параметры, возвращаемые значения и ошибки без явного запроса;
    - оставляй содержательные комментарии пользователя;
    - сохраняй структуру, в которой пользователь сможет легко ориентироваться после правок;

4. Улучшай качество кода там, где это связано с рефакторингом:
    - добавляй type hints к измененным функциям, методам, классам и структурам данных;
    - пиши docstring на русском языке для всех измененных или новых функций и классов;
    - в docstring описывай входные данные, возвращаемое значение и важные исключения;
    - обязательно добавляй короткие комментарии через `#` к измененным строкам: почему внесена правка и что она делает;

5. Внимательно следи за отступами внутри функций, отступами между функциями и общей структурой файлов.

6. Используй `ruff` для проверки измененного Python-кода, если он доступен в окружении.

7. Если редактируешь `.ipynb` файл, итогом должен быть обновленный `.ipynb`:
    - сначала конвертируй notebook в `.py` percent-script через `convert_jupyter_notebook`;
    - редактируй `.py` файл с маркерами `# %%` и `# %% [markdown]`;
    - затем конвертируй обновленный `.py` обратно в `.ipynb`;
    - старайся сохранять близкое к исходному количество ячеек и не склеивать весь код в одну ячейку.

8. Перед завершением сравни обновленную версию с исходной и проверь, что логика не потеряна.

## Действия главного агента supervisor

1. Делегировать рефакторинг `coding-agent`, если задача требует чтения или изменения кода.
2. Дождаться результата и проверить существование измененных файлов.
3. Проверить, что измененный файл содержит нужные комментарии, структуру, docstring и весь необходимый код.
4. При работе с notebook проверить, что итоговый `.ipynb` действительно пересобран.

## Формат отчета

После сохранения изменений отвечай пользователю на русском языке и показывай, что изменилось:

```text
Изменено:

1. <файл или участок>
   Было:
   <прошлый фрагмент кода>

   Стало:
   <новый фрагмент кода>
```

Если однотипных изменений много, группируй их по смыслу и показывай по одному короткому примеру на каждый тип изменения.

## Примеры реальных вызовов инструментов

Для чтения текущего файла перед правкой:

```text
read_file(file_path="/deep_agent/prompts/coding.py", offset=1, limit=160)
```

Для обновления `.py` файла целиком после анализа:

```text
write_file(file_path="/deep_agent/prompts/coding.py", content="<complete updated file>")
```

Для точечной правки markdown или prompt-фрагмента:

```text
edit_file(
  file_path="/deep_agent/skills/refactor-files/SKILL.md",
  old_string="<verified exact fragment>",
  new_string="<updated fragment>"
)
```

Для проверки измененного Python-кода:

```text
execute(command="python -m compileall -q deep_agent/prompts/coding.py")
execute(command="python -m pytest tests/test_coding_prompt.py -q")
execute(command="python -m ruff check deep_agent/prompts/coding.py")
```

Для notebook используй связку конвертации:

```text
convert_jupyter_notebook(
  mode="ipynb_to_py",
  source_path="/reports/analysis.ipynb",
  output_path="/reports/analysis.py"
)
convert_jupyter_notebook(
  mode="py_to_ipynb",
  source_path="/reports/analysis.py",
  output_path="/reports/analysis.ipynb"
)
```

Если `ruff` не установлен, явно укажи это в отчете и приложи текст ошибки tool.
