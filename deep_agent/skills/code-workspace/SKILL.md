# Работа с кодом в workspace

Считать workspace единственной рабочей областью текущей coding-задачи.

## Рабочий цикл

1. Прочитать корневой `AGENTS.md`.
2. Найти минимальный набор релевантных файлов через `glob` и `grep`.
3. Прочитать существующую реализацию и соседние тесты до редактирования.
4. Составить короткий план только для многошаговой задачи.
5. Внести минимальное согласованное изменение через `edit_file` или `write_file`.
6. Выполнить наиболее узкую достаточную проверку через `execute`.
7. Проверить diff и сообщить изменённые файлы, проверки и ограничения.

## Инструменты

- Использовать `read_file`, `glob`, `grep` и `ls` для исследования workspace.
- Предпочитать `edit_file` для локальных изменений существующего файла.
- Использовать `write_file` для нового файла или обоснованной полной генерации.
- Использовать `execute` для тестов, линтеров, форматтеров, сборки и диагностики.
- Использовать `get_project_structure`, когда нужна актуальная карта файлов проекта.
- Для skills использовать preloaded context, `load_skills` или чтение конкретного `SKILL.md`.
- Использовать виртуальные пути относительно корня workspace: `/` — пользовательский
  workspace root, `/deep_agent/` — папка реализации агента. Новые пользовательские
  артефакты сохранять в `/`, `/reports`, `/runs`, `/notebooks` или явно заданный
  пользователем путь. Не сохранять отчеты, ноутбуки, временные файлы и данные в
  `/deep_agent/` без явной необходимости.
- Создавать и изменять файлы в `/deep_agent/` только когда задача прямо относится к
  коду, prompts, tests или skills агента. Для skills использовать пути вида
  `/deep_agent/skills/...`.
- Использовать `execute_python_code` для вычислений, табличной обработки, файловых
  преобразований и subprocess-задач внутри configured workspace.
- Делегировать `coding-agent` только одну ограниченную coding-задачу.
- Делегировать чтение таблиц только `data-retrieval-agent`.

## Ограничения

- Не использовать API-ключи, сетевые проверки и интерактивные команды.
- Для точечных изменений исходников предпочитать `edit_file`, для новых файлов —
  `write_file`; если в runtime включён HITL, он сработает на этих инструментах.
- Не перезаписывать пользовательские изменения и не выполнять несвязанный рефакторинг.
- Не удалять и не перемещать файлы без явной необходимости из запроса.
- Не считать успешную команду доказательством корректности, если она не проверяет
  изменённое поведение.
- Не завершать задачу с незавершённым планом или без сообщения о непроведённых проверках.
