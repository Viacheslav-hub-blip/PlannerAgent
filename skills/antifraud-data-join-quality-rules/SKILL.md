# Правила объединения данных

## Основной ключ CSV-источников

Все дополнительные CSV-источники соединяются по ключу:

`epk_id`

Источники:

- `source_1.csv` — ответы клиентов CSI;
- `source_2.csv` — сегментация и правила;
- `source_3.csv` — суммы транзакций.

# Рекомендуемый тип join

Для построения витрины с полным набором признаков используй:

`inner join`

Это означает, что в итоговую витрину попадут только клиенты, которые есть во всех объединяемых источниках.

Если задача требует сохранить всех клиентов из базового источника, можно использовать другой тип join, но это должно быть явно обосновано.

# Контроль качества объединения

Перед объединением проверь:

- наличие поля `epk_id` во всех источниках;
- пропуски в `epk_id`;
- дубликаты по `epk_id`;
- количество строк до join;
- количество строк после join;
- потерю клиентов после объединения;
- пропуски в ключевых аналитических полях после join.

# Важное ограничение

Если в источнике несколько строк на один `epk_id`, сначала определи корректную бизнес-логику:

- оставить все строки;
- агрегировать до уровня клиента;
- выбрать последнюю запись;
- выбрать максимальную сумму;
- выбрать минимальную сумму;
- рассчитать среднюю сумму;
- посчитать количество ответов, транзакций или правил.

Не предполагай, что `epk_id` уникален во всех источниках, пока это не проверено.

# Связь с антифрод-таблицами

Для связи таблицы сработок с клиентскими источниками сначала нужно определить клиентский идентификатор из сработки.

Для связи с raw event tables используй:

- `event_id`, если он есть в обеих таблицах;
- иначе составной ключ: клиент, дата, время или временное окно, сумма, тип операции и дополнительные реквизиты.
